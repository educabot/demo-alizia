## Test cases

### AreaRepository

| # | Caso | Expected |
|---|------|----------|
| 1 | Create area | Área creada con ID asignado y organization_id correcto |
| 2 | List areas de org 1 | Retorna solo áreas de org 1, con Subjects y Coordinators precargados |
| 3 | List areas de org 2 | Retorna solo áreas de org 2 (aislamiento multi-tenant) |
| 4 | Update area (name + description) | Campos actualizados correctamente |
| 5 | AssignCoordinator | AreaCoordinator creado con area_id y user_id correctos |
| 6 | AssignCoordinator duplicado | Retorna error 409 conflict |
| 7 | Preload Subjects en List | Cada área incluye sus materias asociadas |
| 8 | Preload Coordinators en List | Cada área incluye sus coordinadores asignados |

### SubjectRepository

| # | Caso | Expected |
|---|------|----------|
| 1 | Create subject con area_id válido | Materia creada correctamente |
| 2 | List subjects de org 1 | Retorna solo materias de org 1 |
| 3 | List subjects de org 2 | Retorna solo materias de org 2 (aislamiento multi-tenant) |
| 4 | Create subject con area_id inexistente | Error FK constraint |

### Multi-tenancy

| # | Caso | Expected |
|---|------|----------|
| 1 | Crear área en org 1, listar desde org 2 | No aparece en el listado de org 2 |
| 2 | Crear materia en org 1, listar desde org 2 | No aparece en el listado de org 2 |

## Approach

- Usar una DB PostgreSQL de test (docker-compose o CI service)
- Cada test corre migraciones y limpia datos (transaction rollback o truncate)
- `testify/assert` y `testify/require` para assertions
- Seed de test incluye una segunda organización para validar aislamiento
