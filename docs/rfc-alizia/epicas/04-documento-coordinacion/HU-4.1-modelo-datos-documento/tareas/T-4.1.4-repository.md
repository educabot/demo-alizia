# T-4.1.4: Repository GORM del documento de coordinación

**Historia:** HU-4.1 — Modelo de datos del documento
**Tipo:** Backend
**Estado:** ⬜ Pendiente

---

## Descripción

Implementar el repositorio GORM que satisface `CoordinationProvider`. La complejidad principal está en los preloads anidados para `GetByID` (5 niveles de profundidad) y en las transacciones para `Create` (múltiples tablas en una operación atómica).

## Implementación

Archivo: `src/repositories/coordination_repository.go`

```go
package repositories

import (
    "context"

    "go-alizia/src/core/entities"
    "go-alizia/src/core/providers"
    "gorm.io/gorm"
)

type coordinationRepository struct {
    db *gorm.DB
}

func NewCoordinationRepository(db *gorm.DB) providers.CoordinationProvider {
    return &coordinationRepository{db: db}
}

// GetByID con preloads completos del arbol
func (r *coordinationRepository) GetByID(ctx context.Context, orgID, docID uint) (*entities.CoordinationDocument, error) {
    var doc entities.CoordinationDocument
    err := r.db.WithContext(ctx).
        Where("id = ? AND organization_id = ?", docID, orgID).
        Preload("Area").
        Preload("DocTopics.Topic").
        Preload("Subjects.Subject").
        Preload("Subjects.SubjectTopics.Topic").
        Preload("Subjects.Classes.ClassTopics.Topic").
        First(&doc).Error
    if err != nil {
        return nil, err
    }
    return &doc, nil
}

// List sin preloads profundos (solo resumen)
func (r *coordinationRepository) List(ctx context.Context, orgID uint, areaID *uint) ([]entities.CoordinationDocument, error) {
    var docs []entities.CoordinationDocument
    query := r.db.WithContext(ctx).
        Where("organization_id = ?", orgID).
        Preload("Area")
    if areaID != nil {
        query = query.Where("area_id = ?", *areaID)
    }
    err := query.Find(&docs).Error
    return docs, err
}

// Create con transaccion (documento + junction tables)
func (r *coordinationRepository) Create(ctx context.Context, doc *entities.CoordinationDocument) error {
    return r.db.WithContext(ctx).Transaction(func(tx *gorm.DB) error {
        // 1. Crear documento principal
        if err := tx.Create(doc).Error; err != nil {
            return err
        }
        // GORM crea automaticamente las relaciones anidadas
        // si DocTopics, Subjects, etc. estan populados en el struct
        return nil
    })
}

// Delete con cascade (Postgres maneja el cascade via FKs)
func (r *coordinationRepository) Delete(ctx context.Context, orgID, docID uint) error {
    return r.db.WithContext(ctx).
        Where("id = ? AND organization_id = ?", docID, orgID).
        Delete(&entities.CoordinationDocument{}).Error
}
```

## Notas

- `GetByID` necesita 5 niveles de `Preload` para traer el arbol completo: `DocTopics.Topic`, `Subjects.Subject`, `Subjects.SubjectTopics.Topic`, `Subjects.Classes.ClassTopics.Topic`
- `List` solo hace `Preload("Area")` para mantener la respuesta liviana — el frontend muestra nombre del area y status
- `Create` usa transaccion; GORM puede crear relaciones anidadas automáticamente si los structs estan populados
- `Delete` depende del `ON DELETE CASCADE` de Postgres — al borrar el documento, todas las junction tables se limpian
- `Update` debe usar `db.Save()` o `db.Model().Updates()` según si se quiere actualizar campos selectivos o todo el struct
- Los métodos `AddTopics`, `SetSubjects`, `AddSubjectTopics`, `CreateClasses` y `UpdateClass` siguen el mismo patrón de filtrado por org y uso de transacciones donde sea necesario
- Considerar agregar `Order("class_number ASC")` en el preload de `Classes` para mantener el orden
