# T-4.2.1: Usecase crear documento de coordinación

**Historia:** HU-4.2 — Wizard de creación
**Tipo:** Backend
**Estado:** ⬜ Pendiente

---

## Descripción

Implementar el usecase `CreateDocument` que recibe los datos completos del wizard (nombre, area, fechas, topics, materias con class_count y topics por materia) y crea el documento con todas sus junction tables en una transacción. Debe validar que el area pertenece a la org, que los topics existen al nivel correcto, que las materias pertenecen al area, y que el usuario es coordinador del area.

## Implementación

Archivo: `src/core/usecases/coordination/create_document.go`

```go
package coordination

import (
    "context"
    "fmt"

    "go-alizia/src/core/entities"
    "go-alizia/src/core/providers"
)

type CreateDocumentInput struct {
    Name      string              `json:"name"`
    AreaID    uint                `json:"area_id"`
    StartDate string             `json:"start_date"` // "2026-03-15"
    EndDate   string             `json:"end_date"`
    TopicIDs  []uint             `json:"topic_ids"`
    Subjects  []SubjectInput     `json:"subjects"`
}

type SubjectInput struct {
    SubjectID  uint   `json:"subject_id"`
    ClassCount int    `json:"class_count"`
    TopicIDs   []uint `json:"topic_ids"`
}

type CreateDocumentUsecase struct {
    coordProvider providers.CoordinationProvider
    areaProvider  providers.AreaProvider
    topicProvider providers.TopicProvider
    orgProvider   providers.OrganizationProvider
}

func (uc *CreateDocumentUsecase) Execute(ctx context.Context, orgID, userID uint, input CreateDocumentInput) (uint, error) {
    // 1. Validar que el area existe y pertenece a la org
    area, err := uc.areaProvider.GetByID(ctx, orgID, input.AreaID)
    if err != nil {
        return 0, fmt.Errorf("area no encontrada: %w", err)
    }

    // 2. Validar que el usuario es coordinador del area
    // Verificar en area_coordinators que (area_id, user_id) existe

    // 3. Validar que los topics existen al nivel correcto
    // Leer config.topic_selection_level de la org
    // Verificar que cada topic_id tiene level == topic_selection_level

    // 4. Validar que las materias pertenecen al area
    // Para cada subject en input.Subjects, verificar subject.area_id == input.AreaID

    // 5. Validar que los topic_ids de cada materia son subconjunto de input.TopicIDs

    // 6. Crear documento en transaccion
    doc := &entities.CoordinationDocument{
        OrganizationID: orgID,
        Name:           input.Name,
        AreaID:         area.ID,
        // StartDate y EndDate parseados
        Status: "draft",
        DocTopics: buildDocTopics(input.TopicIDs),
        Subjects:  buildSubjects(input.Subjects),
    }

    err = uc.coordProvider.Create(ctx, doc)
    if err != nil {
        return 0, err
    }

    return doc.ID, nil
}
```

## Notas

- El usecase recibe `orgID` y `userID` del contexto de autenticación (middleware)
- La validación de `topic_selection_level` se lee de `organizations.config` — si la org configura nivel 3 (categories), solo se aceptan topics de ese nivel
- Los `topic_ids` de cada materia deben ser un subconjunto de los `topic_ids` del documento — no se puede asignar a una materia un topic que no fue seleccionado en el paso 1
- El status siempre arranca en `draft` — no se puede crear un documento publicado directamente
- Si alguna validación falla, retornar error descriptivo con código 422
- La transacción es crítica: si falla la creación de una junction table, todo se rollbackea
