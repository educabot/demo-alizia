# T-6.3.1: Tool registry

**Historia:** HU-6.3 — Motor de function calling
**Tipo:** Backend
**Estado:** ⬜ Pendiente

---

## Descripcion

Implementar un registro de tools reutilizable que permita definir herramientas con su nombre, descripcion, parametros (JSON Schema) y handler function. El registry es la pieza central que conecta las definiciones de tools (enviadas al LLM) con las funciones que las ejecutan.

## Implementacion

```go
// src/core/usecases/ai/tool_registry.go
package ai

import (
    "encoding/json"
    "fmt"

    "go-alizia/src/core/providers"
)

// ToolHandler es la funcion que ejecuta una tool call.
// Recibe los argumentos como JSON string, retorna el resultado como string.
type ToolHandler func(args string) (string, error)

// RegisteredTool representa una tool con su definicion y handler.
type RegisteredTool struct {
    Definition providers.ToolDefinition
    Handler    ToolHandler
}

// ToolRegistry mantiene un set de tools para un contexto especifico.
type ToolRegistry struct {
    tools map[string]RegisteredTool
}

// NewToolRegistry crea un registry vacio.
func NewToolRegistry() *ToolRegistry {
    return &ToolRegistry{tools: make(map[string]RegisteredTool)}
}

// Register agrega una tool al registry.
// Si ya existe una tool con el mismo nombre, se sobreescribe.
func (r *ToolRegistry) Register(def providers.ToolDefinition, handler ToolHandler) {
    r.tools[def.Name] = RegisteredTool{Definition: def, Handler: handler}
}

// GetDefinitions retorna todas las definiciones de tools registradas.
// Este slice se envia al LLM como parte del request.
func (r *ToolRegistry) GetDefinitions() []providers.ToolDefinition {
    defs := make([]providers.ToolDefinition, 0, len(r.tools))
    for _, t := range r.tools {
        defs = append(defs, t.Definition)
    }
    return defs
}

// Execute ejecuta el handler de una tool por nombre.
// Retorna error si la tool no existe en el registry.
func (r *ToolRegistry) Execute(name string, args string) (string, error) {
    tool, ok := r.tools[name]
    if !ok {
        return "", fmt.Errorf("unknown tool: %s", name)
    }
    return tool.Handler(args)
}

// HasTool verifica si una tool esta registrada.
func (r *ToolRegistry) HasTool(name string) bool {
    _, ok := r.tools[name]
    return ok
}

// Count retorna la cantidad de tools registradas.
func (r *ToolRegistry) Count() int {
    return len(r.tools)
}
```

## Comportamiento

1. Se crea un `ToolRegistry` nuevo para cada contexto (coord doc chat, lesson plan chat, etc.)
2. Se registran las tools especificas del contexto via `Register()`
3. `GetDefinitions()` retorna el slice de definiciones para enviar al LLM junto con los mensajes
4. Cuando el LLM responde con `tool_calls`, el loop (T-6.3.2) llama a `Execute()` con el nombre y argumentos
5. Si la tool no existe → retorna error `unknown tool: X` (el loop lo maneja como tool result)

## Notas

- El registry es **context-specific** — cada chat context crea su propia instancia con sus propias tools
- Las tools se registran al inicializar el usecase, no de forma global
- `Execute` es el metodo que llama el tool execution loop (T-6.3.2) cuando recibe `tool_calls` del LLM
- Se reutiliza `providers.ToolDefinition` definido en HU-6.1 para evitar duplicar tipos
- `HasTool` y `Count` son helpers para tests y validacion
